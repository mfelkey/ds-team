Three rules the system never breaks

6. When It Asks You Something

At six points in the pipeline, the system stops and waits for you. You'll get a text message and an email, and the terminal will show what it needs.

What to do when it stops:

  1. Open the file it tells you about (e.g., dev/requirements/PROJ-XXXXX_PRD.md)

  3. Go back to the terminal and type APPROVE or REJECT

7. When Things Go Wrong

ğŸ”‡ Agent sits there doing nothing (30+ minutes)

What happened: The LLM hit a timeout or ran out of context window space.

Fix: Press Ctrl+C. Check that Ollama is still running (curl http://localhost:11434/api/tags). Re-run the same agent â€” it's safe, it just starts over.

ğŸ” Agent keeps repeating the same text over and over

What happened: The model lost its "stop" instruction. This is the most common failure with local LLMs on large outputs.

Fix:

  1. Press Ctrl+C. The file may have good content before the loop started.
  2. Open the output file and delete the repeated content.
  3. To prevent it next time, open the agent's Python file and add max_tokens=4096 to the Task() constructor, and add an "OUTPUT DISCIPLINE" instruction to the agent's backstory. Full instructions: agents/mobile/output_guard_patch.md

ğŸ“„ Agent exits immediately with "No project context found"

What happened: There's no logs/PROJ-*.json file.

Fix: Run python agents/orchestrator/classify.py first.

ğŸ“„ Agent exits with "Missing TIP" or "Missing PRD" or similar

What happened: A required upstream document hasn't been created yet.

Fix: Run the agents in order. Check which one was skipped.

ğŸ¤– Output has // TODO or placeholder code

What happened: The model took shortcuts instead of writing complete code.

Fix: Open the agent's Python file, find the Task(description=...) block, and add: "No placeholders. No TODO comments. Every function must be complete and working." Then re-run.

ğŸ’¥ Output is about the wrong project or drifted off-topic

What happened: The upstream artifact excerpt was too short and the model lost context.

Fix: In the agent's run_* function, find the line like f.read()[:3000] and increase the number (try [:5000] or [:8000]). Then re-run.

ğŸ”Œ Ollama connection refused

sudo systemctl start ollama # Verify: curl http://localhost:11434/api/tags

ğŸ’¾ Out of memory (Ollama crashes or machine freezes)

What happened: The model is too large for your available RAM.

Fix: Don't run strategy agents and build agents at the same time â€” they use different models. Run the full planning phase first, then the build phase. If the 72b model is too big, drop back to the 32b: edit TIER1_MODEL in config/.env.

ğŸ”„ How to start over

ğŸ”„ How to re-run one agent

8. Change Models, Tune Agents, Add New Ones

Switch to a different LLM

Two lines in config/.env control which models the entire team uses:

TIER1_MODEL=ollama/qwen2.5:72b # Planning agents (the thinkers) TIER2_MODEL=ollama/qwen2.5-coder:32b # Build agents (the coders)

To change: pull the new model (ollama pull <name>), update the line, done. No code changes.

Make an agent behave differently

Every agent's behavior is controlled by text you can edit â€” no coding required.

What to changeWhere to find itExample
The agent's personality and expertiseAgent(backstory="...") in the build_* functionAdd "You always prefer serverless architectures" to the Architect's backstory
What the agent is asked to produceTask(description="...") in the run_* functionAdd a new section to the PRD template: "10. REGULATORY REQUIREMENTS"
How much upstream context the agent seesf.read()[:3000] in the run_* functionChange [:3000] to [:6000] to give it more context
Stop the agent from loopingAgent backstory + Task constructorAdd "OUTPUT DISCIPLINE: stop after all sections are complete" to backstory, add max_tokens=4096 to Task

Add a new agent

Copy any existing agent file (e.g., backend_developer.py) and modify it. Every agent follows the same pattern:

  1. A build_* function that returns a CrewAI Agent with role, goal, backstory, and model
  2. A run_* function that creates a Task, runs it in a Crew, saves the output, and updates the project context
  3. A __main__ block that loads the project context and finds upstream artifacts
The key pattern: Every agent reads upstream artifacts from the artifacts array in the project context JSON. It finds them by type (e.g., "PRD", "TAD"). When it finishes, it appends its own artifact to the same array. That's how agents find each other's work.
